✅ Pro (Pattern di design riusciti)

PRO Director-driven convergence: _director_agenda fissa per ogni round il punto conteso + assignments per-partecipante → cross-talk forzato, niente monologhi paralleli. Pattern replicato da 0xNyk Council.
PRO Notazione RE/WHY/DECISION: output ultra-sintetico (3 righe telegrafiche) → massima informazione, zero prosopopea. Anti-allucinazione esplicito ("ZERO hallucinations", "prefix assumptions with ASSUME:").
PRO Secretary live: consolidamento strategia DOPO OGNI SINGOLO TURN (non solo a fine round) → i modelli successivi costruiscono sopra, non ricominciano. Pattern ripreso da Manus e adottato in tavolarotonda.
PRO Memory guardrail: memguard.plan_warmup legge /proc/meminfo + Ollama /api/tags, esclude correttamente i REMOTE_PEERS dal warmup (i remoti non hanno RAM), warmup in 80% MemAvailable.
PRO Heterogeneous models: 8 modelli locali + 6 remoti free + Claude + Gemini = superficie di disaccordo genuina. Stessa filosofia di "panel-of-experts-awscao".
PRO Resume con start_round: parametro che permette di riprendere un dibattito interrotto dalla numerazione corretta → fault tolerance.
PRO Web fetch contestuale: se il topic contiene URL, fetch_context le scarica (Playwright) e inietta il contenuto come contesto PER TUTTI i peer (inclusi text-only).
PRO Bridge verso Claude Code: implementation_md finale è un brief pronto per essere dato in pasto a Claude Code CLI → chiude il loop dibattito→codice.

❌ Contro (svantaggi confermati)

CONTRO Niente privacy tier esplicito: il codice non distingue tra dati che possono uscire verso Groq/Cerebras e dati che devono restare locali. Un utente può inavvertitamente inviare PII a vendor free-tier. (Risolto in tavolarotonda con privacy_tier + redact_for_privacy.)
CONTRO Niente Problem Restate Gate: il topic è esposto direttamente ai partecipanti senza riformulazione preventiva. Se 3 agenti riformulano diversamente → avviso di problema mal posto (0xNyk Council). (Aggiunto in tavolarotonda come phase_restate.)
CONTRO Niente Adversarial evidence retrieval: il web fetch è solo "fonte unica" (lista URL), non distingue supporting vs counter-evidence. (Aggiunto in tavolarotonda come adversarial_research.)
CONTRO Niente minority report: il piano finale è solo "analysis.md + implementation.md", senza la sezione esplicita "caso CONTRO la decisione". (Aggiunto in tavolarotonda come palace.minority_report.)
CONTRO Niente Open Questions in cima al verdict: la sintesi non mette in evidenza ciò che il panel NON è riuscito a rispondere. (0xNyk Council: "What the council can't answer matters more than where it agrees".) Risolto in tavolarotonda.

⚠️ Criticità (bug e problemi da risolvere)

#1 CRITICITÀ ALTA Prompt injection via directives: _panel_prompt inserisce le direttive utente con marker "MODERATOR INSTRUCTIONS (override the original topic if in conflict)". Un topic compromesso (es. URL malevolo che finisce nel web_context, o direttiva con injection) può dirottare l'intero panel. Nessuna validazione/sanificazione. Fix tavolarotonda: sanitize_directive() con rimozione di 14 injection markers + troncamento a 500 char.
#2 CRITICITÀ ALTA Niente timeout esplicito su _say: run_bare_model + gemini_complete + claude_query non hanno timeout. Un modello in deadlock blocca tutto il roundtable (indefinitamente). Fix tavolarotonda: timeout 120s default per modello + retry 3x con backoff esponenziale + circuit breaker.
#3 CRITICITÀ MEDIA _injections in-memory non persistente: _injections: dict[str, list[str]] = {} vive in RAM; restart di agent-core perde tutte le direttive post-resume. Fix tavolarotonda: Memory Palace JSON persistente su disco (resume trasparente).
#4 CRITICITÀ ALTA (compliance) Privacy leak a Groq/Cerebras free tier: REMOTE_PEERS sono chiamati via LiteLLM con topic + transcript completi. Nessuna redazione PII prima dell'invio. Se l'utente inserisce dati sensibili nel topic → leak. Fix tavolarotonda: tier free_api_ok con redact_for_privacy() (email, IP, telefono IT, CF).
#5 BASSA Niente rate limit per round: l'utente può chiedere 20 round senza feedback intermedio. Fix tavolarotonda: env-var configurabile + early-stop su converged.
#6 BASSA Niente retry/backoff su remoti: fail rete = messaggio [error: ...] nel transcript. Fix tavolarotonda: 3 tentativi con backoff esponenziale (1s, 2s, 4s).
#7 BASSA _project_tree con os.walk depth 2: lento su progetti grandi. Fix futuro: git ls-files + .gitignore respect.
#8 BASSA _extract_json_obj depth naive: non gestisce } dentro stringa JSON. Fix parziale in tavolarotonda: try/except su tutti gli estrattori.
#9 BASSA MAX_ROUNDS = 20 hardcoded: no override via env/config. Fix tavolarotonda.
#10 BASSA Temperature 0.85 hardcoded per tutti i model/remote/worker. Fix tavolarotonda: per-agent configurabile.

⚠️ Minority Report (caso CONTRO)

Il pattern Director-driven è robusto MA può creare echo chamber controllata: se il Director (modello singolo) sbaglia focus per 2 round consecutivi, l'intero dibattito deraglia. Soluzione: implementare dissenting director (un secondo modello che produce un focus alternativo) e fare media pesata. Il problema attuale è che il Director è un SINGLE POINT OF FAILURE nonostante la pipeline abbia N modelli. Aggiungere una fase "director review" ogni 3 round per validare che il focus corrente sia ancora rilevante.
